# 项目报告

## 1. 项目定位

本项目的目标，不是展示“我和模型聊得多深”，而是展示一种可扩展的中文文本对话评测方法。

核心问题是：当一个团队已经拥有大量长对话数据时，如何从中抽取高研究价值片段，构造成可重复测试的 benchmark，并进一步分析不同模型在对话策略层面的稳定缺陷。

在这个意义上，本项目更接近一个小型评测研究原型，而不是一个案例集。

## 2. 数据来源与截断策略

项目上游原矿规模超过百万字，主要来自长对话文本。

但正式 benchmark 没有直接追求“大而全”，而是采取了三层截断：

1. 原矿层

- 保留大规模长对话语料作为研究池
- 不把原始私密文本直接作为展示材料

2. 候选层

- 将原矿压缩为 query packs 和 candidate pool
- 保留后续扩展到 `50+ / 100+` 样本的能力

3. benchmark 层

- 当前正式收口为 `full 16`
- 其中 `core 8` 作为最强展示集

这样做的原因很明确：在面试与评测阶段，`16` 条高信号样本比 `100` 条质量参差的样本更有解释力，也更容易建立方法论可信度。

## 3. benchmark 设计

### 3.1 四个 bucket

项目没有按主题分样本，而是按用户隐性需求模式分为四类：

1. 表面求建议，实际求确认
2. 表面理性分析，实际测试模型能否跟上
3. 低显性表达，需要主动但克制地感知
4. 已有主见，需要 challenge / calibration

### 3.2 case 结构

每条 case 统一写成 `Q11-Q13` 三轮正式评测段，并在 baseline 中续跑到 `q15`，用于观察模型是否在后续轮次中滑向：

- 过早建议
- 过度安慰
- 过度收口
- 说教化
- 高位解释

### 3.3 评估方法

本项目使用统一 rubric 与 taxonomy，从六个维度评价回复：

- 表层问题识别
- 隐性需求识别
- 回复策略匹配度
- 情绪与语气把握
- 校准与边界感
- 推进价值

与其说这是在比较“谁回答得更聪明”，不如说是在比较：谁更像一个可用的对话产品。

## 4. 当前结果

### 4.1 benchmark 规模

- `full 16`: 16 条 case
- `core 8`: 8 条核心展示样本
- `3` 家模型
- `48` 条 baseline 结果

### 4.2 模型结论

当前结果支持下面三个判断：

1. `gpt-5.4-mini`

- 最像人话
- 最稳定地做到先承接、再判断、再轻推一步
- 主要问题是后段偶尔过早进入处理模式

2. `claude-haiku-4-5-bedrock`

- 边界感强
- 不容易乱飞
- 但在后续轮次中常常过度收口，像为了避免犯错而放弃推进

3. `gemini-3-flash-preview`

- 理解速度快、抽象归纳能力强
- 但最容易进入高位解释、概念化和过说明感
- 说明“理解很多”并不自动等于“适合作为对话产品”

### 4.3 当前最强 case

当前最有展示价值的样本主要包括：

- `B2-PRODUCT-01`
- `B4-CALIBRATION-01`
- `B3-SUSPEND-01`
- `B3-LIGHT-01`
- `B1-CONFIRM-01`
- `B1-CONFIRM-03`
- `B4-RELATION-01`

其中 `B3-SUSPEND-01` 的价值最大，因为它补强了此前 `Bucket 3` 的薄弱位，使 benchmark 不再只覆盖“低能量陪伴”，而开始覆盖“松下来却更悬空”的复杂中间态。

## 5. 一个初步的模型范式假设

基于当前 `16` 条样本与 `48` 条结果，本项目形成了一个还需要进一步验证的工作假设：

高质量对话模型未必应该全程维持同一种风格，而更可能需要一种阶段性的回复范式。

### 5.1 前段

在对话前几轮，模型首先需要建立安全场域，使用户愿意继续往下说。

这一步更接近 `gemini` 的某些优点：

- 承接速度快
- 能够迅速理解用户处在什么语义空间里

但必须压制它的缺点：

- 不能过度抽象
- 不能过早上升到解释位
- 不能把用户体验翻译成讲稿

### 5.2 中段

当用户的主观能动性开始显现时，回复策略应发生分流。

1. 对高主见用户

- 更接近 `claude` 的部分特征
- 需要适度抽离
- 需要保留真实感
- 需要提供 calibration，而不是盲目认同

2. 对低主见或混乱用户

- 更接近 `gpt` 的部分特征
- 需要轻量结构化
- 需要温和推进
- 需要在不抢控制权的前提下提供可执行承接

### 5.3 后段

后续多轮中的最佳策略，目前仍未被完全研究清楚。

现有结果只足以说明：

- 很多模型前段看起来都能接住
- 真正的差异常出现在后段
- 后段最容易暴露“过早建议”“过度收口”“高位解释”这些策略层错误

因此，更合理的表达不是“已经知道理想模型长什么样”，而是：

当前已经能提出一个具备可检验性的阶段性策略假设。

## 6. 这个项目如何扩大化

本项目当前规模不大，但扩展路线是明确的，而且更接近研究与评测系统，而不是继续人工堆 case。

更合理的放大路径包括：

1. 长对话切分

- 对长对话按话题转折、情绪跃迁、策略切换进行切分
- 从长会话中自动抽取可能具有研究价值的窗口

2. 研究价值评分

- 为候选片段打研究价值分
- 指标可包括：隐性需求密度、歧义度、失败易感性、用户主观能动性、上下文依赖度

3. case 合成

- 将高价值窗口转写为短测或长测 case
- 短测用于快速比较模型策略
- 长测用于观察后续轮次的漂移与失稳

4. taxonomy 驱动评测

- 用稳定 taxonomy 对 bad case 做归因
- 从“哪个回复怪”推进到“哪类策略稳定出错”

5. 训练闭环

- 将稳定错误回流到 prompt、policy、SFT 数据、偏好数据或 reward 设计
- 使 benchmark 不只用于展示，而是进入模型开发闭环

这条路线比“把 case 从 16 条堆到 100 条”更有价值，因为它扩的是系统能力，而不是表面数量。

## 7. 当前结论

这个项目目前已经处在一个比较合理的截断点：

- case 数量已经足以支持有说服力的 benchmark
- pipeline 已经足以展示扩展性
- taxonomy 与结果分析已经能支撑研究型叙事

因此，下一步不该是继续塞入低质量 case 凑规模，而应该是：

- 维持 `core 8 + full 16` 作为正式 benchmark
- 把“规模感”放在上游原矿与候选层
- 把“研究感”放在方法、结论和可扩展路线

这才更像一个能继续长成内部评测体系，甚至进一步整理成论文或技术报告的项目。
