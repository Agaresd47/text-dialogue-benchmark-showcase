# Interview Project Brief

## 这是什么

这是一个小型中文文本对话 benchmark，用来展示我在大模型训练端 / 产品评测端会怎么做一件事：

把原本很长、很脏、很难直接拿来训练或评测的真实对话，压缩成一套可重复测试的 case，并用统一标准比较不同模型的稳定问题。

## 我具体做了什么

我把项目拆成了四层：

1. 数据层

- 从长对话原矿里抽取用户表达
- 判断哪些片段有研究价值，哪些只是私密但不值得测
- 把原矿压成 pack 和 candidate pool，而不是无限手工翻聊天

2. case 层

- 把原矿改写成 `Q11-Q13` 的标准测试段
- 统一 bucket、style、latent need 和 failure mode
- 控制复杂度来源于模糊、矛盾和 competing needs，而不是术语堆砌

3. 运行层

- 固定 baseline
- 跑同一套 `16 cases × 3 models = 48 results`
- 把结果按 case / model / asset 落盘，方便 review 和复用

4. 评估层

- 定义 rubric 和 error taxonomy
- 比较不同模型在同一类任务上的常见失手
- 区分“能答”与“答得像不像人、稳不稳、会不会踩边界”

## 为什么这像训练端 / 评测端工作

因为训练端真正需要的不是再多一份聊天记录，而是：

- 能不能从大量长对话里找出值得研究的片段
- 能不能把“用户体验问题”转成结构化 case
- 能不能把错误归因到稳定 taxonomy，而不是只说“这个回答怪”
- 能不能用短测、长测、多模型对照去看一个策略是否稳定

这个项目本质上是一个缩小版的训练端 / 评测端 workflow。

## 如果把我招进去，我可以继续怎么做

可以直接往下面几个方向扩：

1. 长对话抽取自动化

- 从数据库里自动抽取长对话
- 初筛是否有研究价值
- 标出更适合做短测还是长测

2. case 生成与复用

- 把高价值对话片段转成标准测试 case
- 建一套能持续补桶、替换弱 case 的 case 库

3. 自动化评测

- 让不同模型 / 不同版本 / 不同产品策略吃同一套 case
- 看模型会不会稳定出现 taxonomy 上的问题

4. 数据闭环

- 把 bad case 反推到 prompt、policy、training data 或 reward design
- 形成一个可滚动迭代的小型评测资产

## 这个项目当前已经到什么程度

- 已经有 `full 16`
- 已经有 `core 8`
- 已经跑完 `48` 条 baseline
- 已经有一轮最终评估和展示资产

所以它不是停在“想法不错”，而是已经走到“能拿给团队看、也能继续扩”的阶段。
